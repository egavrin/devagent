# packages/runtime

This file provides guidance to AI agents working in this component.

## Purpose and Scope

- **component**: `packages/runtime`
- **purpose**: Shared internal platform for configuration, credentials, sessions, skills, task loop logic, review helpers, and tool integrations used by the CLI and executor.
- **primary languages/platforms**: TypeScript, Bun, Node.js

## Important Paths and Interfaces

```text
src/core/                    # Config, errors, sessions, credentials, instruction and skill loading
src/engine/                  # Task loop, judges, review helpers, prompt assets
src/tools/                   # Built-in tools, registry, git safety, LSP and MCP integrations
src/index.ts                 # Public package exports
src/engine/prompts/*.md      # Prompt assets copied during build
src/**/*.test.ts             # Component tests
package.json                 # Build/test scripts and exports
```

## Build, Test, and Run

```bash
cd packages/runtime
bun run build
bun run typecheck
bun run test
```

## Architecture and Workflows

- This package is the consolidated replacement for the older split `core`/`engine`/`tools` workspace model. Treat `src/core`, `src/engine`, and `src/tools` as internal subdirectories of one package.
- Changes to exported types, config, or shared utilities can cascade into every other workspace package. Keep exports intentional and verify cross-package compile health.
- `src/core/skills/` contains runtime source code for skill discovery and loading. It is not the same thing as repo-local guidance in `.agents/skills/`.
- `src/engine/prompts/*.md` are build inputs and must stay consistent with the task loop and tests when modified.
- Tooling code under `src/tools/` is frequently security-sensitive because it touches shell execution, filesystem operations, Git state, LSP, and MCP behavior.

## Generated Files, Conventions, and Pitfalls

- Do not edit `dist/`; it is generated by `bun run build`.
- Follow fail-fast behavior rigorously here. Silent fallbacks in runtime code tend to create cross-package regressions that are difficult to diagnose later.
- Prefer adding focused tests next to the affected module before changing behavior. Runtime changes often need both unit coverage and adjacent integration coverage.
- Keep package boundaries intact: `runtime` must not import from `cli`, `executor`, `providers`, or `arkts`.

## Local Skills

- No component-local skill directory exists here. Use the shared repo skills from `.agents/skills`, especially `review`, `security-checklist`, `debug-test-failure`, `tdd-workflow`, and `testing`.
